RELATÓRIO: POR QUE O OPEN HANDLE (TCPWRAP) AINDA OCORRE

1) RESUMO RÁPIDO
- Sintoma: Jest com --detectOpenHandles reporta um handle TCPWRAP relacionado a chamadas de teste que usam `request(app).get('/ready')`.
- Objetivo: eliminar o open handle sem alterar a rota `/ready`.

2) AÇÕES EXECUTADAS (resumido)
- Inspecionei e localizei o pool MySQL em `backend/src/shared/database/connection.js` (exporta pool mysql2/promise).
- Adicionei validações e handlers de fechamento em área de setup/teardown de testes:
  - `backend/tests/setup/globalTeardown.js` criado para fechar `pool.end()` ao final da suíte inteira.
  - Removi encerramentos por arquivo conflitantes no `jest.setup.js` e deixei teardown global.
- Para o arquivo de teste afetado (`tests/integration/fase4-encerramento.spec.js`): criei/abri um servidor dedicado (`app.listen(0)`) e alterei os requests do `supertest` para usar esse server; adicionei rastreamento e destruição de sockets e fechamentos explícitos no `afterAll` (incluindo `pool.end()` local no file-level afterAll como fallback).
- Adicionei um handler `process.on('beforeExit', ...)` como fallback para tentar garantir fechamento do pool antes do processo encerrar.

3) COMANDOS EXECUTADOS (principais)
- `npm run lint`
- `npm test` (rodou todas as suites)
- `npx jest tests/integration/fase4-encerramento.spec.js --detectOpenHandles --runInBand` (várias execuções durante o diagnóstico)

5) POR QUE O PROBLEMA PERMANECE (diagnóstico técnico)
- Observado: o handle reportado pelo Jest é um socket TCP ainda aberto quando o processo tenta sair.
- Possíveis fontes:
  - conexões MySQL não fechadas (pool) — mitigadas por `pool.end()` em globalTeardown e por fechamentos explícitos por arquivo; porém, nem sempre elimina o TCPWRAP mostrado;
  - servidor HTTP de testes (servidor criado por supertest internamente) deixando sockets de conexão em KEEP-ALIVE — mitigado criando server dedicado e destruindo sockets rastreados antes do close; ainda assim Jest reportou TCPWRAP apontando a `app.listen` em execuções específicas;
  - outro código em runtime que abre sockets TCP e os mantém (monitor, cliente, pool interno do mysql2, ou sockets de healthcheck do container quando testes executados contra DB local).

6) O QUE FOI TESTADO / RESULTADO
- depois de aplicar: globalTeardown, per-file pool.end(), server tracking + destroy, keepAliveTimeout = 0, beforeExit handler — executei `npx jest ... --detectOpenHandles` repetidas vezes.
- Resultado observável: testes passam, mas em algumas execuções Jest ainda reportou `TCPWRAP` apontando para `app.listen` linha do arquivo de teste (ou para o próprio supertest). Em tentativas finais, após destruição explícita de sockets e fechamento do pool dentro do próprio `afterAll` do arquivo, o problema ficou intermitente; em algumas execuções ainda apareceu.

7) CONCLUSÃO TÉCNICA
- O problema é causado por handles TCP abertos (sockets) que não foram fechados antes do processo de testes encerrar.
- Foram aplicadas múltiplas medidas corretivas (fechamento do pool no globalTeardown e no afterAll, destruição de sockets rastreados, desativar keepAlive, unref, beforeExit fallback). Essas medidas reduziram ocorrências, mas não eliminaram completamente a detecção do `TCPWRAP` em todas as execuções no ambiente atual.

8) AÇÕES RECOMENDADAS (próximos passos práticos)
- Executar a suite com `--detectOpenHandles` em ambiente controlado (sem background processes/IDEs) e capturar a stack completa do handle para identificar o stack exacto (por ex. rodar com `--runInBand --detectOpenHandles --logHeapUsage` e repetir até obter stack estável).
- Instrumentar código para logar quando `pool.end()` é chamado e quantas conexões ativas restam (expor `pool._allConnections.length` etc.) para confirmar que o pool foi completamente drenado.
- Verificar se algum outro teste/module abre conexões (externas) que não são encerradas (ex.: clientes TCP, healthchecks, conexões Docker internas).
- Como atalho, se aceitável, usar `--forceExit` (não recomendado), ou garantir que o processo de testes invoque `globalTeardown` e que nenhum servidor seja iniciado sem ser fechado.
